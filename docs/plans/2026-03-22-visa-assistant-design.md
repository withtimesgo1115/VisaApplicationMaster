# Visa Assistant Design Proposal (Discussion Draft)

## 1) 背景与目标

当前仓库处于**绿地阶段**（仅有 `README.md` 和 `LICENSE`），尚未形成既有架构与实现。

产品目标：面向中国公民的海外签证 DIY 场景，提供：

1. 选择目的地后，快速返回申请材料清单与申请步骤；
2. 对用户上传材料做“是否齐全”的检查；
3. 材料不齐全时，明确提示缺失项与准备指引；
4. 在材料齐全且风险可控时，支持“一键发起申请/提交流程”；
5. 在申请前采集关键背景信息（职业、收入、资产等）用于个性化指导；
6. 同时支持微信小程序、网页端，并预留原生 App 演进路径。

---

## 2) 设计原则

- **可信优先**：签证要求变化快，规则可信度优先于“自动化炫技”。
- **分阶段落地**：先做可运行的 MVP（人机协同），再增强自动化和平台覆盖。
- **规则可解释**：每个“缺材料”判断都要有可读原因。
- **合规与最小化采集**：仅采集必要信息，敏感数据全链路保护。
- **平台策略务实**：先做最能触达中国用户的入口，再扩展原生端。

---

## 3) 三种可行架构路线（对比）

### 方案 A（推荐）：微信小程序 + 运营后台 Web + 模块化单体后端

**适配场景**：小团队、快速验证、重运营闭环。

**优点**：
- 微信生态触达强，获客和使用门槛低；
- 后端单体便于快速迭代业务规则；
- 运营后台可做人审、改规则、补材料沟通；
- 在“材料不齐全”场景下可迅速形成服务闭环。

**缺点**：
- 小程序端交互能力和原生端相比有约束；
- 需要设计好“后续扩展到原生 App”的边界。

---

### 方案 B：统一跨端前端框架（如 Taro/uni-app）+ 统一后端

**适配场景**：团队希望较高代码复用，且能接受跨端细节差异调优成本。

**优点**：
- 理论上多端复用率高；
- 前端团队协作成本较低（相对多套端并行）。

**缺点**：
- 小程序、Web、App 三端细节差异仍需大量适配；
- 复杂表单、上传链路、平台 API 差异容易侵蚀复用收益。

---

### 方案 C：Web + 原生 App + 小程序各自独立前端 + 后端服务化

**适配场景**：预算充足、团队规模较大、目标是中长期高并发与复杂生态接入。

**优点**：
- 每端可做到最佳体验；
- 服务拆分后可独立扩展。

**缺点**：
- 初期投入最大，交付周期长；
- 对目前阶段（从 0 到 1）风险偏高。

---

## 4) 推荐结论

推荐先落地**方案 A**：

- **MVP 平台面**：微信小程序（用户侧）+ Web 运营后台（审核/规则管理/客服协同）；
- **后端形态**：模块化单体（清晰领域边界），避免过早微服务；
- **演进路径**：当小程序端体验/性能成为明确瓶颈时，再补原生 App。

原因：当前最核心风险不是“端能力不够炫”，而是**规则准确性、材料完整性判断、运营处理效率**。

---

## 5) MVP 功能边界（第一阶段）

### 用户侧（小程序）

1. 目的地与签证类型选择；
2. 个人背景信息采集（职业、收入、资产、旅行史等）；
3. 动态材料清单展示（按国家/签证类型/用户画像）；
4. 材料上传与状态跟踪（已上传/待补齐/已通过）；
5. 缺失项与准备指导（模板、注意事项、示例）；
6. 提交“审核准备完成”请求。

### 运营侧（Web）

1. 签证规则管理（版本、来源、生效时间、审核状态）；
2. 工单/案件审核（自动判定结果 + 人工复核）；
3. 一键回传补充要求；
4. 审核通过后进入“可提交”状态；
5. 操作审计与日志追踪。

### 暂不纳入 MVP

- 全自动无人工干预提交；
- 覆盖所有国家所有签证细分；
- 重度 OCR 智能抽取作为硬依赖（可后置增强）。

---

## 6) 领域划分（Bounded Context）

1. **Identity & Consent**：身份、登录、隐私同意、授权记录；
2. **Visa Requirement Catalog**：签证要求规则库（来源、版本、审核）；
3. **Applicant Profile & Intake**：申请人画像与申请意图数据；
4. **Document Collection & Completeness**：材料上传、校验、完整性判定；
5. **Case Review & Submission Ops**：人工审核、状态流转、提交编排；
6. **Notification & Audit**：通知、审计、可追溯记录。

---

## 7) 关键流程设计（MVP）

1. 用户选择目的地 + 签证类型；
2. 系统引导填写背景信息；
3. 规则引擎生成个性化材料清单；
4. 用户上传材料；
5. 自动完整性检查（规则 + 基础文件质量检查）；
6. 不完整则返回“缺失清单 + 准备建议”；
7. 完整后进入人工复核；
8. 审核通过则标记“可提交”，触发提交流程（或人工代提）。

---

## 8) 数据与规则模型（概念层）

### 核心实体

- `Destination`：国家/地区、签证类型集合；
- `VisaRuleVersion`：规则版本、来源 URL、生效时间、审核人、置信状态；
- `ApplicantProfile`：职业、收入、资产、旅行史、家庭结构等；
- `ApplicationCase`：申请主单据（状态机）；
- `RequiredDocumentItem`：需提交项、判定条件、说明、模板；
- `UploadedDocument`：文件元数据、质量评分、关联材料项；
- `CompletenessResult`：完整性结果、缺失项、理由、建议；
- `ReviewDecision`：人工复核结论与备注。

### 状态机（示意）

`Draft -> CollectingDocs -> AutoChecked(Complete/Incomplete) -> HumanReview -> ReadyToSubmit -> Submitted/Closed`

---

## 9) “一键申请”定义（避免误解）

MVP 阶段建议将“一键申请”定义为：

- 在系统内完成材料齐套校验与人工复核后，
- 一键触发**标准化提交编排流程**（包括外部平台提交任务、人工代办任务或半自动操作流）。

这样既满足“用户感知的一键”，也规避外部签证系统差异带来的不确定性。

---

## 10) 风险与缓解

1. **签证规则变更频繁**
   - 缓解：规则版本化 + 来源追溯 + 定期复检机制 + 内部审核发布流。

2. **完整性误判（误报/漏报）**
   - 缓解：规则可解释、支持人工覆盖、记录误判回流用于规则迭代。

3. **敏感信息与文件安全**
   - 缓解：传输/存储加密、最小权限、访问审计、短时链接、分级留存与销毁策略。

4. **运营瓶颈**
   - 缓解：案件分级、标准化审核模板、高频问题自动化提示。

---

## 11) 分阶段路线图

### Phase 1（MVP，6-10 周）
- 小程序用户主流程 + Web 审核后台 + 规则库基础能力；
- 覆盖少量高频国家/签证类型；
- 形成“采集-检查-补件-复核-提交”的闭环。

### Phase 2（增强，8-12 周）
- 增加国家覆盖与规则细分；
- 强化文档质量检测与引导；
- 引入更细粒度自动化编排。

### Phase 3（扩展）
- 评估原生 App（若小程序受限明确）；
- 视吞吐与稳定性拆分高压模块（如文档处理、通知中心）。

---

## 12) 这版讨论需要你确认的 3 个关键点

1. MVP 是否采用**Hybrid（自动检查 + 人工复核）**而非完全自助？
2. 平台优先级是否确定为**小程序优先，Web 运营并行，原生 App 后置**？
3. “一键申请”在 MVP 中是否同意定义为“**一键触发提交流程编排**”而非“100% 全自动外部提交”？
